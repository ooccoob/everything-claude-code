# TDD 指令

此指令呼叫 **tdd-guide** Agent 来強制执行测试驱动开发方法论。

## 此指令的功能

1. **建立接口骨架** - 先定义类型/接口
2. **先产生测试** - 撰寫失敗的测试（RED）
3. **实作最小程序代码** - 撰寫剛好足以通过的程序代码（GREEN）
4. **重构** - 在测试保持绿色的同时改进程序代码（REFACTOR）
5. **验证覆盖率** - 确保 80% 以上测试覆盖率

## 何时使用

在以下情况使用 `/tdd`：
- 实作新功能
- 新增新函数/元件
- 修复 Bug（先撰寫重现 bug 的测试）
- 重构现有程序代码
- 建构关键商业逻辑

## 运作方式

tdd-guide Agent 会：

1. **定义接口**用於输入/输出
2. **撰寫会失敗的测试**（因为程序代码还不存在）
3. **执行测试**并验证它们因正确的原因失敗
4. **撰寫最小实作**使测试通过
5. **执行测试**并验证它们通过
6. **重构**程序代码，同时保持测试通过
7. **检查覆盖率**，如果低於 80% 则新增更多测试

## TDD 循环

```
RED → GREEN → REFACTOR → REPEAT

RED:      撰寫失敗的测试
GREEN:    撰寫最小程序代码使其通过
REFACTOR: 改进程序代码，保持测试通过
REPEAT:   下一个功能/情境
```

## TDD 最佳实务

**应该做：**
- ✅ 在任何实作前先撰寫测试
- ✅ 在实作前执行测试并验证它们失敗
- ✅ 撰寫最小程序代码使测试通过
- ✅ 只在测试通过后才重构
- ✅ 新增边界情况和错误情境
- ✅ 目标 80% 以上覆盖率（关键程序代码 100%）

**不应该做：**
- ❌ 在测试之前撰寫实作
- ❌ 跳过每次变更后执行测试
- ❌ 一次撰寫太多程序代码
- ❌ 忽略失敗的测试
- ❌ 测试实作细节（测试行为）
- ❌ Mock 所有東西（优先使用整合测试）

## 覆盖率要求

- **所有程序代码至少 80%**
- **以下类型需要 100%：**
  - 财务计算
  - 验证逻辑
  - 安全关键程序代码
  - 核心商业逻辑

## 重要提醒

**強制要求**：测试必须在实作之前撰寫。TDD 循环是：

1. **RED** - 撰寫失敗的测试
2. **GREEN** - 实作使其通过
3. **REFACTOR** - 改进程序代码

绝不跳过 RED 阶段。绝不在测试之前撰寫程序代码。

## 与其他指令的整合

- 先使用 `/plan` 理解要建构什麼
- 使用 `/tdd` 帶著测试实作
- 如果发生构建错误，使用 `/build-fix`
- 使用 `/code-review` 审查实作
- 使用 `/test-coverage` 验证覆盖率
